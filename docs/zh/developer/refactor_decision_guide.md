# 重构决策指南

> 适用于个人主导决策、由 Agent 协助实现的长期演化型项目。

## 为什么需要这份指南

本项目已经经历了多轮真实的结构调整，例如响应解析、并行处理、验证器规则、日志与稳定性防护。  
这些重构大多不是为了“看起来更优雅”，而是因为某个功能落地困难、某个顽固 bug 反复出现，或者同类问题开始持续复发。

因此，本指南的目标不是鼓励“为了整洁而重构”，而是帮助区分：

- 什么时候继续打补丁更划算
- 什么时候已经进入结构性问题，应该重构
- 什么时候虽然闻到了坏味道，但还不值得动地基

## 核心原则

### 1. 重构是解决结构性问题，不是审美活动

如果问题的根源在模块职责、数据流、失败处理、统一入口、状态收口等结构层面，那么继续追加补丁通常只会提高后续维护成本。

### 2. 只有能说清病灶，才启动重构

不要因为“这里很乱”就开始重构。  
更好的触发句式是：

- 失败状态和成功状态共用记录逻辑，导致 checkpoint 误判
- 响应解析散落在多个 handler，导致同类脏数据要修很多次
- 游戏特定规则混在通用逻辑里，导致修 A 游戏时影响 B 游戏
- UI 更新和验证日志直接耦合，导致错误洪峰时前端卡死

如果能把问题描述到这个粒度，通常就说明重构目标已经足够明确。

### 3. 补丁和重构都可以是正确选择

不是所有坏味道都值得立刻重构。  
关键不在于“乱不乱”，而在于：

- 问题是否重复出现
- 修复是否经常跨多个模块
- 当前结构是否持续制造新问题
- 本次变更是否已经超过局部补丁的收益边界

## 什么时候优先补丁，而不是重构

以下情况通常先补丁更划算：

- 问题只出现一次，且根因明确
- 修复范围局限在一个模块内
- 不需要引入新抽象也能稳定收口
- 该区域近期还会有更大变动，先用小修补止血更合理
- 当前没有足够证据证明这是结构性问题

### 典型信号

- 改 1 到 2 个函数即可修复
- 没有出现跨层修改
- 不会影响同类问题的其他入口
- 修复后能直接补一条测试，且逻辑边界清晰

## 什么时候应该考虑重构

以下情况通常说明问题已经从“代码错误”升级为“结构错误”：

- 同一类 bug 连续出现两次以上
- 一个需求落地时总要同时改很多不相干文件
- 修 bug 依赖大量隐性前提，只能靠记忆维持
- 某个模块已经让人“不敢动”
- 局部补丁开始互相打架，修一处炸三处
- 某类失败状态没有统一收口点，导致行为不一致

### 典型信号

- 同类逻辑分散在多个 handler / service / workflow 中
- 不同游戏或不同 provider 的特殊规则混在通用代码里
- 成功、失败、重试、回退逻辑分别写在不同地方
- 为了修一个 bug，不得不理解过多无关上下文
- 新功能接入时总要绕过已有抽象，临时开旁路

## 什么时候即使闻到坏味道，也先别重构

以下情况要谨慎：

- 只是命名不好看
- 只是结构不够“优雅”
- 没有复发证据
- 还说不清真正病灶在哪里
- 当前正在交付关键功能，重构会显著扩大风险面
- 这次只是想“顺手整理一下”

一句话：  
**没有明确病灶的重构，通常只是把不确定性搬到别处。**

## 启动重构前，先写 3 行

每次决定重构前，先写出以下内容：

### 1. 这次只解决什么问题

例如：

- 让响应解析从脆弱规则修补改成结构化解析
- 让失败批次和成功批次进入统一结果收集逻辑
- 让游戏特定验证规则从通用验证器中分离

### 2. 什么结果算完成

例如：

- 所有 handler 统一走新 parser
- 旧实现只保留兼容壳，不再作为主入口
- 同类错误的修复入口从多个位置收敛到一个位置

### 3. 这次明确不做什么

例如：

- 不顺手重写整个 API 抽象层
- 不顺手调整前端 UI
- 不顺手改动无关 provider

这一步能有效防止“修一个点，最后重构半个系统”。

## 对 Agent 任务的推荐写法

对于 Agent，最怕的是：

- “这里有点乱，整理一下”
- “顺便优化一下结构”
- “感觉这里该重构了”

更有效的任务写法是：

- 当前问题是什么
- 结构病灶在哪里
- 允许修改哪些文件
- 不允许碰哪些模块
- 验收标准是什么

### 推荐模板

```text
目标：收敛某类重复 bug 的入口
病灶：失败状态记录分散在多个模块，导致 checkpoint 误判
允许修改：相关 handler、service、validator、测试
不要改：无关 UI、无关 provider、发布脚本
验收：失败文件不会被标记为完成；现有成功路径不回退；补充回归测试
```

## 对长期维护最有价值的重构方向

本项目更适合优先重构这些区域：

- 解析链路
- 验证链路
- checkpoint / progress / archive
- 并发调度
- 日志、报告与故障可观察性

原因很简单：  
这些区域不是“功能锦上添花”，而是整个系统的防御工事。  
它们一旦结构失控，后续所有功能开发都会变得更脆弱。

## 不要把重构变成默认模式

对长期项目来说，最危险的不是没有重构，而是：

**一遇到阻力就默认进入重构。**

更好的顺序应该是：

1. 先判断这是单点 bug，还是结构问题
2. 如果是单点 bug，先补丁
3. 如果是结构问题，再定义病灶和边界
4. 控制重构范围，只动当前必须收敛的部分
5. 用测试、日志或报告证明重构真的解决了原问题

## 最后一条判断标准

当你开始觉得：

- “怎么修都不对劲”
- “每次都要记住很多隐藏前提”
- “这块代码已经没人敢碰”
- “补丁成本已经高于问题本身”

那通常就该停止继续缝补，转而修复产生问题的结构。

这不是重构冲动，而是成熟的维护判断。
