# 增量更新 MVP 状态清单

## MVP Done

- 归档上传链路已打通：项目现有译文可上传到 `mods_cache.sqlite`，作为增量更新基线。
- 旧归档 schema 已兼容迁移：修复了 `source_versions.snapshot_hash` 全局唯一和 `source_entries` 仅按 `version_id + entry_key` 去重的问题。
- 归档读取已按 `file_path + key` 工作，避免同名文件或重复 key 串档。
- 增量更新工作流已拆分为快照、差异、准备、翻译、构建、归档几个 service，`update_translate.py` 只负责主流程编排。
- 增量更新默认仅使用归档中真实存在的目标语言，不再错误展开为游戏支持的全部语言。
- 源文件扫描已按项目 `source_language` 过滤，只扫描源语言目录，不再误扫假本地化文件。
- Dry-run 预扫描已可正常返回摘要。
- 非 dry-run 已在真实项目“商船”上跑通，能正确继承旧译文并仅翻译变更条目。
- 前端项目历史页的 `archive_update`/`import` I18N 已修复，增量更新页签 `History` 已本地化。
- 增量更新页增加了轮询兜底，避免 WebSocket 状态未送达时前端无限转圈。

## Known Issues

- 增量更新页面当前仍依赖“WebSocket + 轮询兜底”的双通道状态同步；根因未完全收敛到单一机制。
- 本地模型对极小文本也可能发生过度推理，翻译质量和成本控制仍需继续约束。
- 历史代码和 legacy 分支仍较多，归档、校对、增量更新之间的数据契约虽然已基本拉齐，但还没有完全制度化。
- 目前未启用“官方复用”与“全局复用”；当前 MVP 只支持“旧译文继承 + 变更重翻”。
- 真实项目回归测试仍偏少，现有测试主要覆盖关键服务层和最小集成路径。
