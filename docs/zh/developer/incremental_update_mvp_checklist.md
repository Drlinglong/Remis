# 增量更新 MVP 验收 Checklist

本清单用于在真实项目上验证“增量更新”MVP 是否稳定可用。

---

## 一、 验收前置条件

- 项目已成功创建，且 `source_path` 可访问。
- 项目 `source_language` 已正确设置。
- 项目至少已有一份可用译文目录。
- 已执行过一次“上传翻译”，并成功建立归档基线。
- 当前待测试的新源码目录中，至少包含一处可识别的文本变更。

---

## 二、 归档基线验收

### 1. 上传翻译

- [ ] 点击“上传翻译”后不再出现 500。
- [ ] 项目历史中出现“存档更新”记录。
- [ ] 历史描述显示为本地化文本，而不是原始 key 或英文旧描述。

### 2. 归档内容有效

- [ ] `check-archive` 能检测到存档点。
- [ ] 页面显示的版本号和最后更新时间正常。
- [ ] 默认目标语言只显示归档中真实存在的语言。

---

## 三、 预扫描验收

### 3. 源语言过滤

- [ ] 预扫描只读取项目 `source_language` 对应目录。
- [ ] 不会把假本地化文件误当作源文件参与比较。

### 4. Dry-run 摘要

- [ ] 预扫描能够正常结束，不无限转圈。
- [ ] 页面能进入“预扫描摘要”步骤。
- [ ] `summary.total` 与源文件条目数一致。
- [ ] `summary.changed` 与实际修改条目数一致。
- [ ] `summary.unchanged` 与旧译文可继承条目数一致。

### 5. 日志与状态同步

- [ ] 执行日志区域可以滚动。
- [ ] 预扫描完成后，前端能正确结束 loading 状态。
- [ ] WebSocket 异常时，轮询兜底仍能推进 UI。

---

## 四、 正式执行验收

### 6. 增量翻译执行

- [ ] 正式执行后，只有 `new/changed` 条目进入翻译。
- [ ] `unchanged` 条目不重复调用翻译。
- [ ] 执行结束后页面显示“增量翻译完成”。

### 7. 输出重建

- [ ] 输出目录为 `Remis_Incremental_Update`。
- [ ] 输出文件结构与新版源文件模板一致。
- [ ] 未变化条目继承旧译文。
- [ ] 变化条目被新译文正确覆盖。

### 8. 归档回写

- [ ] 正式执行后会生成新的归档版本。
- [ ] 下一次预扫描会以前一次执行结果为基线。
- [ ] 校对面板仍能读取归档内容。

---

## 五、 最小真实项目用例

建议固定保留一个最小验收项目，例如：

- 项目源文件：1 个
- 总条目：8 条
- 改动条目：1 条
- 目标语言：`zh-CN`

目标结果：

- [ ] 预扫描结果：`total=8, new=0, changed=1, unchanged=7`
- [ ] 正式执行成功
- [ ] 输出文件可人工核对
- [ ] 新归档版本可被下一轮增量更新识别

---

## 六、 当前不纳入 MVP 验收范围

以下内容暂不作为 MVP 必须项：

- 官方译文复用
- 全局复用
- 自动巡检 / watchlist
- Steam API 自动检查更新
- 多项目批量增量更新

---
